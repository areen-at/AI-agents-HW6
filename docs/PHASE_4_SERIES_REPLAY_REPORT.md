## Implemented behavior

- Series controller is independent of MCP, models, Gmail, UI, and deployment.
- Production series requires exactly six valid sub-games.
- Each series receives one unique series ID.
- Each valid game slot receives one unique sub-game ID.
- Every attempt receives a unique attempt ID.
- Attempt number and valid-game index are tracked separately.
- Technical failures invalidate attempts without counting or scoring them.
- Invalid attempts remain in result metadata and event logs.
- Replacement attempts continue until a valid sub-game completes or the configured safety limit is
  reached.
- Infrastructure/application failures are never converted into strategic game losses.
- Scores aggregate only from valid terminal sub-game records.
- Events use schema version `1.0`.
- Events include IDs, timestamp, seed, action validation, state hashes, terminal result, score, and
  accepted action sequence where applicable.
- Event payloads are redacted for common secret/token/credential key names.
- Event logs and reports are written through atomic temp-file replacement.
- Offline replay uses only seed, IDs, accepted role/action sequence, and game settings.
- Replay verification checks the final state hash before a sub-game is accepted.
- Internal report contains group metadata, students, GitHub URL, owned MCP URLs, timezone,
  `sub_games`, invalid attempts, and calculated totals.
- `reports/internal_game_report.json` can be generated by:

  ```powershell
  python main.py --mode internal --config config.json --engine-only
  ```
